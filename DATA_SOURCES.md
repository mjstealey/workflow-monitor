## 1. braindump.yml — Workflow Metadata

**Source file:** `<submit_dir>/braindump.yml`
**When read:** Once at startup
**Module:** `braindump.py` — `load_braindump()` → `WorkflowInfo` dataclass
**Why it matters:** This is the bootstrap file that tells the monitor where everything else lives. Without it, no other data source can be located.

### Fields

| Field | Type | Description | Why needed |
|---|---|---|---|
| `wf_uuid` | str | Unique workflow identifier (UUID4) | Primary key correlating all data sources; used to filter stampede.db queries and tag JSONL events |
| `root_wf_uuid` | str | Root workflow UUID (for nested/sub-workflows) | Identifies the top-level workflow when sub-workflows are present |
| `dax_label` | str | Human-readable workflow label (e.g., "diamond") | Displayed in TUI header and event log metadata |
| `submit_dir` | Path | Workflow submit directory path | Root for locating stampede.db, kickstart output, condor logs, and JSONL output |
| `user` | str | Username who submitted the workflow | Displayed in TUI; used to match against `condor_userprio` entries |
| `planner_version` | str | Pegasus planner version (e.g., "5.0.7") | Displayed in TUI header and analytics |
| `dag_file` | str | DAG filename (e.g., "diamond-0.dag") | Used to derive stampede.db path: `<dag_file>.stampede.db` |
| `condor_log` | str | HTCondor event log filename | Used to derive condor log path for event monitoring |
| `timestamp` | str | Workflow planning/submission timestamp | Displayed in workflow metadata |
| `basedir` | Path | Workflow base directory (defaults to `submit_dir` parent) | Used for file resolution |

### Derived Paths (computed properties on `WorkflowInfo`)

| Property | Derivation | Purpose |
|---|---|---|
| `stampede_db` | `submit_dir / f"{dag_file.replace('.dag','')}.stampede.db"` | Primary data source path |
| `jobstate_log` | `submit_dir / "jobstate.log"` | Alternative job state source (not currently used) |
| `condor_log_path` | `submit_dir / condor_log` | HTCondor event log |
| `dag_path` | `submit_dir / dag_file` | DAG file for reference |
| `dagman_out_path` | `submit_dir / f"{dag_file}.dagman.out"` | DAGMan output log |

### Reference Documentation

- **Pegasus braindump format:** Generated by `pegasus-plan`; documented in the [Pegasus User Guide — Monitoring and Debugging](https://pegasus.isi.edu/documentation/user-guide/monitoring-debugging.html)
- **Local reference:** `~/.claude/references/pegasus.md` — Section: "Workflow Planning and Submission"

## 2. stampede.db (SQLite) — Workflow State & Job Tracking

**Source file:** `<submit_dir>/<dag_stem>.stampede.db`
**When read:** Every `--interval` seconds (default 2s)
**Module:** `db.py` — `StampedeDB` class → `WorkflowSnapshot` dataclass
**Why it matters:** This is the **primary data source** for the monitor. It's a SQLite database maintained by `pegasus-monitord` that records every workflow and job state transition in real time. All job progress, timing, and state information comes from here.

### Connection Details

- Opened read-only (`?mode=ro` URI parameter) to avoid interfering with pegasus-monitord writes
- 5-second timeout for lock contention
- Queries are filtered by `wf_id` (resolved from `wf_uuid` at startup) for multi-workflow databases
- Graceful degradation: `sqlite3.OperationalError` (locked DB) returns a minimal empty snapshot

### Tables and Queries

#### `workflow` table — Workflow identity

**Query:** `_resolve_wf_id()`
```sql
SELECT wf_id FROM workflow WHERE wf_uuid = ?
```

| Column | Type | Description |
|---|---|---|
| `wf_id` | int | Internal integer ID (used as FK in all other tables) |
| `wf_uuid` | str | UUID matching braindump.yml |

**Why:** Maps the UUID from braindump.yml to the integer `wf_id` used as a foreign key throughout the database.

#### `workflowstate` table — Workflow lifecycle

**Query 1:** `get_workflow_state()` — Current workflow state
```sql
SELECT state, timestamp, status
FROM workflowstate
WHERE wf_id = ?
ORDER BY timestamp DESC
LIMIT 1
```

**Query 2:** `get_workflow_times()` — Start/end timestamps
```sql
SELECT state, timestamp
FROM workflowstate
WHERE wf_id = ?
ORDER BY timestamp
```

| Column | Type | Description | Why needed |
|---|---|---|---|
| `state` | str | `WORKFLOW_STARTED` or `WORKFLOW_TERMINATED` | Determines if workflow is active, complete, or unknown |
| `timestamp` | float | Unix timestamp of state transition | Used to compute elapsed time |
| `status` | int | Exit status (0 = success, non-zero = failure) | Determines success/failure outcome |

#### `job` table — Job roster

| Column | Type | Description | Why needed |
|---|---|---|---|
| `job_id` | int | Internal job identifier (FK for job_instance, task) | Primary key for joining across tables |
| `exec_job_id` | str | DAG node name (e.g., "preprocess_ID0000001") | Human-readable job identifier used in TUI and diagnostics |
| `type_desc` | str | Job type classification | Distinguishes compute jobs from infrastructure |
| `wf_id` | int | Foreign key to workflow table | Filters jobs belonging to this workflow |

**Job types (`type_desc` values):**
| Value | Category | Description |
|---|---|---|
| `compute` | User work | Actual science/compute jobs defined in the workflow |
| `stage-in-tx` | Infrastructure | File staging from input sources to compute nodes |
| `stage-out-tx` | Infrastructure | File staging from compute nodes to output destinations |
| `create-dir` | Infrastructure | Directory creation on execution sites |
| `stage-worker` | Infrastructure | Worker package staging for remote sites |
| `cleanup` | Infrastructure | Cleanup of intermediate/temporary files |
| `registration` | Infrastructure | Output file registration in the Replica Catalog |

#### `job_instance` table — Job execution attempts

| Column | Type | Description | Why needed |
|---|---|---|---|
| `job_id` | int | FK to job table | Links instance to logical job |
| `job_instance_id` | int | Unique instance identifier | FK for jobstate and invocation queries |
| `job_submit_seq` | int | Retry sequence number (0, 1, 2, ...) | `MAX(job_submit_seq)` identifies the latest attempt |
| `exitcode` | int | Raw exit code (wait status format) | Converted to real exit code for display/diagnostics |
| `site` | str | Execution site/resource name | Shown in TUI job detail and analytics |
| `stdout_file` | str | Relative path to job stdout | Used to locate kickstart output for diagnostics |
| `stderr_file` | str | Relative path to job stderr | Used for failure diagnostics |

**Note on `exitcode`:** The raw value is in wait-status format. Values > 128 are decoded as `raw >> 8` to get the actual exit code. The `real_exitcode()` function in `db.py` handles this conversion.

#### `jobstate` table — Job state transitions

| Column | Type | Description | Why needed |
|---|---|---|---|
| `job_instance_id` | int | FK to job_instance | Links state to specific execution attempt |
| `state` | str | Pegasus job state | Drives the TUI job status display |
| `timestamp` | float | Unix timestamp of transition | Used for timing, event ordering, and incremental queries |
| `jobstate_submit_seq` | int | Ordering within instance | Disambiguates same-timestamp transitions |

**Job state flow (typical success path):**
```
SUBMIT → EXECUTE → JOB_TERMINATED → JOB_SUCCESS → POST_SCRIPT_STARTED → POST_SCRIPT_TERMINATED → POST_SCRIPT_SUCCESS
```

**State mapping to display categories** (defined in `_STATE_MAP`):

| Raw Pegasus State | Display State | Meaning |
|---|---|---|
| `POST_SCRIPT_SUCCESS`, `JOB_SUCCESS` | SUCCESS | Job completed successfully |
| `POST_SCRIPT_FAILURE`, `JOB_FAILURE` | FAILED | Job failed |
| `EXECUTE` | RUNNING | Job is actively executing |
| `SUBMIT` | QUEUED | Job submitted to HTCondor, awaiting matchmaking |
| `PRE_SCRIPT_STARTED`, `PRE_SCRIPT_SUCCESS` | PRE | Pre-script phase |
| `POST_SCRIPT_STARTED`, `POST_SCRIPT_TERMINATED` | POST | Post-script phase |
| `JOB_TERMINATED` | DONE | Job terminated (pre-post-script) |
| `JOB_HELD`, `JOB_EVICTED` | HELD | Job held or evicted from execution |
| *(no state yet)* | UNSUBMITTED | Job exists in DAG but not yet submitted |

#### `task` table — Transformation metadata

| Column | Type | Description | Why needed |
|---|---|---|---|
| `job_id` | int | FK to job table | Links task metadata to job |
| `transformation` | str | Transformation name (e.g., "preprocess") | Used as human-readable display name for compute jobs |
| `argv` | str | Command-line arguments | Shown in job detail and diagnostics |

#### `invocation` table — Kickstart invocation records

| Column | Type | Description | Why needed |
|---|---|---|---|
| `job_instance_id` | int | FK to job_instance | Links invocation metrics to execution attempt |
| `task_submit_seq` | int | ≥ 0 for real invocations; -1 for pre/post-scripts | Filter: `task_submit_seq >= 0` for actual job invocations |
| `maxrss` | int | Peak resident set size in KB | Memory usage tracking; `MAX(maxrss)` per job shown in TUI |

### Main Query: `get_jobs()`

The primary query uses a CTE (`latest_ji`) to find the latest execution attempt per job, then joins across `job`, `job_instance`, `jobstate`, `task`, and `invocation` tables. Key aspects:
- Latest attempt: `MAX(job_submit_seq)` per `job_id`
- Current state: Most recent `jobstate` entry for that instance (by timestamp and submit_seq DESC)
- Timing: `MIN(timestamp) WHERE state='SUBMIT'` for submit_time, `MIN WHERE state='EXECUTE'` for start_time, `MAX WHERE state IN (JOB_TERMINATED, JOB_SUCCESS, JOB_FAILURE)` for end_time
- Memory: `MAX(invocation.maxrss) WHERE task_submit_seq >= 0`

### Incremental Query: `get_events_since(after_ts)`

Used by the JSONL event logger to capture only new state transitions since the last poll:
```sql
SELECT j.exec_job_id, j.type_desc, js.state, js.timestamp,
       j.job_id, ji.exitcode, ji.stdout_file, ji.stderr_file,
       MAX(inv.maxrss) AS maxrss
FROM job j
JOIN job_instance ji ON j.job_id = ji.job_id
JOIN jobstate js ON ji.job_instance_id = js.job_instance_id
WHERE js.timestamp > ?
ORDER BY js.timestamp ASC, js.jobstate_submit_seq ASC
```

### Output: `WorkflowSnapshot` dataclass

| Field | Type | Description |
|---|---|---|
| `wf_state` | str | Current workflow state |
| `wf_status` | Optional[int] | Exit status |
| `wf_start` | Optional[float] | Workflow start timestamp |
| `wf_end` | Optional[float] | Workflow end timestamp |
| `jobs` | List[JobRecord] | All jobs with current state |
| `recent_events` | List[Dict] | Last N state transitions |
| `poll_time` | float | When this snapshot was captured |

**Computed properties on `WorkflowSnapshot`:** `is_running`, `is_complete`, `succeeded`, `failed`, `elapsed`, `job_counts()`, `done_count()`, `failed_count()`, `held_count()`, `running_count()`, `queued_count()`, `progress_pct()`, `compute_jobs()`, `infra_jobs()`, `held_jobs()`, `failed_jobs()`

### Output: `JobRecord` dataclass

| Field | Type | Description |
|---|---|---|
| `job_id` | int | Internal ID |
| `exec_job_id` | str | DAG node name |
| `type_desc` | str | Job type |
| `raw_state` | Optional[str] | Raw Pegasus job state |
| `exitcode` | Optional[int] | Raw exit code (wait status) |
| `site` | Optional[str] | Execution site |
| `submit_time` | Optional[float] | SUBMIT state timestamp |
| `start_time` | Optional[float] | EXECUTE state timestamp |
| `end_time` | Optional[float] | Terminal state timestamp |
| `transformation` | Optional[str] | Transformation name |
| `task_argv` | Optional[str] | Command arguments |
| `stdout_file` | Optional[str] | Stdout file path |
| `stderr_file` | Optional[str] | Stderr file path |
| `maxrss` | Optional[int] | Peak RSS in KB |

**Computed properties:** `disp_state` (display category), `duration` (seconds, live for RUNNING), `is_compute`, `display_name` (prefers transformation for compute, else exec_job_id)

### Reference Documentation

- **Pegasus stampede schema:** Maintained by `pegasus-monitord`; schema documented in [Pegasus Stampede Schema Reference](https://pegasus.isi.edu/documentation/reference-guide/stampede-schema-reference.html)

## 3. HTCondor Queue (condor_q) — Live Job Status

**Source:** HTCondor Schedd daemon
**Access method:** Python bindings (`htcondor.Schedd.query()`) with subprocess fallback (`condor_q -json`)
**When polled:** Every `--interval` seconds (default 2s)
**Module:** `htcondor_poll.py` — `query_queue()` → `List[Dict]` (ClassAd dicts)
**Why it matters:** Provides real-time job status, resource requests, hold reasons, and transfer progress that are not available in stampede.db. The stampede.db records Pegasus-level state transitions; condor_q shows the HTCondor-level view with richer detail.

### Subprocess Command

```bash
condor_q -json -attributes ClusterId,ProcId,JobStatus,Cmd,RemoteHost,QDate,JobStartDate,DAGNodeName,Owner,HoldReason,HoldReasonCode,RequestCpus,RequestMemory,RequestDisk,RequestGpus,ImageSize,NumJobStarts,AccountingGroup,TransferInputSizeMB,BytesSent,BytesRecvd [-name <SCHEDD>] [-constraint <EXPR>]
```

| Attribute | Type | Description | Why needed | HTCondor Reference |
|---|---|---|---|---|
| `ClusterId` | int | HTCondor cluster (job batch) ID | Primary job identifier in HTCondor | [condor_submit manual — ClusterId](https://htcondor.readthedocs.io/en/latest/man-pages/condor_submit.html) |
| `ProcId` | int | Process ID within cluster (for multi-proc jobs) | Distinguishes individual processes in a cluster | Same as above |
| `JobStatus` | int | Numeric job status code | Core status indicator | [Job ClassAd Attributes — JobStatus](https://htcondor.readthedocs.io/en/latest/classad-attributes/job-classad-attributes.html#JobStatus) |

| `RemoteHost` | str | Slot name where job is executing (e.g., "slot1@machine") | Shows execution location; used in JSONL fingerprinting | [Job ClassAd Attributes — RemoteHost](https://htcondor.readthedocs.io/en/latest/classad-attributes/job-classad-attributes.html#RemoteHost) |

### JobStatus Codes

| Code | Status | Description |
|---|---|---|
| 1 | Idle | Waiting for matchmaking |
| 2 | Running | Actively executing |
| 3 | Removed | Removed from queue |
| 4 | Completed | Finished execution |
| 5 | Held | Held due to error or policy |
| 6 | TransferOutput | Transferring output files |
| 7 | Suspended | Suspended by admin/policy |

### Access Method Details

1. **Python bindings (preferred):** Tries `htcondor` then `htcondor2` modules via `Schedd.query(projection=..., constraint=...)`
2. **Subprocess fallback:** `condor_q -json -attributes ...` with 10-second timeout
3. **Credential support:** Token path (`_CONDOR_SEC_TOKEN_DIRECTORY`), GSI cert/key (`X509_USER_CERT`/`X509_USER_KEY`), password file (`_CONDOR_PASSWORD_FILE`), collector host (`_CONDOR_COLLECTOR_HOST`)

### JSONL Event Fingerprinting

The event logger emits an `htcondor_poll` event only when the queue fingerprint changes. The fingerprint is a frozenset of tuples:
```
(ClusterId, JobStatus, HoldReason, RemoteHost, BytesSent, BytesRecvd)
```
This captures status changes, host assignments, hold events, and transfer progress while suppressing duplicate events when the queue is stable.

## 4. HTCondor History (condor_history) — Post-Completion Metrics

**Source:** HTCondor Schedd history database
**Access method:** Python bindings (`Schedd.history()`) with subprocess fallback (`condor_history -json`)
**When polled:** Throttled — every ~10 seconds (3-5x the base poll interval)
**Module:** `htcondor_poll.py` — `query_history()` → `List[Dict]` (ClassAd dicts)
**Why it matters:** Provides post-completion metrics not available in condor_q (which only shows active jobs) or stampede.db (which lacks HTCondor-level resource usage). CPU/memory efficiency, queue wait time, disk usage, and transfer volumes all come from here.

### Subprocess Command

```bash
condor_history -json -attributes ClusterId,ProcId,DAGNodeName,Owner,Cmd,JobStatus,ExitCode,RemoteWallClockTime,RemoteUserCpu,RemoteSysCpu,CumulativeRemoteUserCpu,RequestCpus,RequestMemory,RequestDisk,RequestGpus,ImageSize,DiskUsage,LastRemoteHost,BytesSent,BytesRecvd,NumJobStarts -match 200 [-name <SCHEDD>] [-constraint <EXPR>]
```

### ClassAd Attributes

| Attribute | Type | Description | Why needed | HTCondor Reference |
|---|---|---|---|---|
| `ClusterId` | int | HTCondor cluster ID | Job identity; history fingerprinting | See condor_q section |
| `ProcId` | int | Process ID within cluster | Job identity | See condor_q section |
| `DAGNodeName` | str | DAG node name | Correlates to Pegasus jobs | See condor_q section |
| `Owner` | str | Job owner | Filtering | See condor_q section |
| `Cmd` | str | Job executable | Display | See condor_q section |
| `JobStatus` | int | Final job status | Outcome classification | See condor_q section |
| `ExitCode` | int | Job exit code | Exit code display in history panel | [Job ClassAd Attributes — ExitCode](https://htcondor.readthedocs.io/en/latest/classad-attributes/job-classad-attributes.html#ExitCode) |
| `RemoteWallClockTime` | float | Total wall-clock time on execute node (seconds) | CPU efficiency denominator; job duration from HTCondor perspective | [Job ClassAd Attributes — RemoteWallClockTime](https://htcondor.readthedocs.io/en/latest/classad-attributes/job-classad-attributes.html#RemoteWallClockTime) |
| `RemoteUserCpu` | float | User-mode CPU time consumed (seconds) | CPU efficiency numerator | [Job ClassAd Attributes — RemoteUserCpu](https://htcondor.readthedocs.io/en/latest/classad-attributes/job-classad-attributes.html#RemoteUserCpu) |
| `RemoteSysCpu` | float | System-mode CPU time (seconds) | Total CPU accounting | [Job ClassAd Attributes — RemoteSysCpu](https://htcondor.readthedocs.io/en/latest/classad-attributes/job-classad-attributes.html#RemoteSysCpu) |
| `CumulativeRemoteUserCpu` | float | Total user CPU across all retries (seconds) | Aggregate CPU accounting for retried jobs | [Job ClassAd Attributes — CumulativeRemoteUserCpu](https://htcondor.readthedocs.io/en/latest/classad-attributes/job-classad-attributes.html#CumulativeRemoteUserCpu) |
| `RequestCpus` | int | CPUs requested | CPU efficiency denominator (multi-core jobs) | See condor_q section |
| `RequestMemory` | int | Memory requested (MB) | Memory efficiency denominator | See condor_q section |
| `RequestDisk` | int | Disk requested (KB) | Resource tracking | See condor_q section |
| `RequestGpus` | int | GPUs requested | Resource tracking | See condor_q section |
| `ImageSize` | int | Peak memory footprint (KB) | Memory efficiency numerator | See condor_q section |
| `DiskUsage` | int | Disk used during execution (KB) | Disk utilization tracking | [Job ClassAd Attributes — DiskUsage](https://htcondor.readthedocs.io/en/latest/classad-attributes/job-classad-attributes.html#DiskUsage) |
| `LastRemoteHost` | str | Final execution slot name | Host distribution analytics | [Job ClassAd Attributes — LastRemoteHost](https://htcondor.readthedocs.io/en/latest/classad-attributes/job-classad-attributes.html#LastRemoteHost) |
| `BytesSent` | int | Total bytes sent to execute node | Transfer volume analytics | See condor_q section |
| `BytesRecvd` | int | Total bytes received from execute node | Transfer volume analytics | See condor_q section |
| `NumJobStarts` | int | Number of execution attempts | Retry counting: `sum(NumJobStarts - 1)` | See condor_q section |

### Derived Metrics

| Metric | Formula | Why |
|---|---|---|
| **CPU efficiency** | `RemoteUserCpu / (RemoteWallClockTime * RequestCpus)` | Measures how well the job utilized allocated CPU cores. Low efficiency (< 50%) suggests I/O-bound workload or over-provisioned CPU requests |
| **Memory efficiency** | `ImageSize / (RequestMemory * 1024)` | Measures actual vs. requested memory. Low efficiency means wasted reservation; > 100% means the job used more than requested |
| **Queue wait time** | `JobStartDate - QDate` | Time from submission to execution start. Long waits indicate resource contention, priority issues, or requirement mismatches |

### JSONL Event Fingerprinting

The history fingerprint is a frozenset of `ClusterId` values. A new `htcondor_history` event is emitted only when a new ClusterId appears, indicating a newly completed job.

### Caching

History results are accumulated across poll cycles. The full accumulated set is passed to `compute_workflow_stats()` at workflow completion.

## 5. HTCondor Pool Status (condor_status) — Resource Inventory

**Source:** HTCondor Collector/Startd daemons
**Access method:** Python bindings (`Collector.query(AdTypes.Startd)`) with subprocess fallback (`condor_status -json`)
**When polled:** Throttled — every ~15 seconds (5-7x the base poll interval)
**Module:** `htcondor_poll.py` — `query_slots()` → `Optional[PoolSummary]`
**Why it matters:** Provides the only view of pool-level resource capacity. Without this, the monitor cannot explain *why* jobs are idle (no slots? all slots busy? resource mismatch?). Pool data is essential for the why-idle diagnostic and the TUI pool panel.

### Subprocess Command

```bash
condor_status -json -attributes Name,Machine,SlotType,Cpus,Memory,Disk,TotalSlotCpus,TotalSlotMemory,TotalLoadAvg,Activity,State,OpSys,Arch,GPUs [-pool <HOST>]
```

### Slot ClassAd Attributes

| Attribute | Type | Description | Why needed | HTCondor Reference |
|---|---|---|---|---|
| `Name` | str | Slot name (e.g., "slot1@machine.example.com") | Unique slot identity | [Machine ClassAd Attributes — Name](https://htcondor.readthedocs.io/en/latest/classad-attributes/machine-classad-attributes.html#Name) |
| `Machine` | str | Machine hostname | Unique machine counting | [Machine ClassAd Attributes — Machine](https://htcondor.readthedocs.io/en/latest/classad-attributes/machine-classad-attributes.html#Machine) |
| `SlotType` | str | `Partitionable`, `Dynamic`, or `Static` | Critical for correct resource aggregation (see below) | [Machine ClassAd Attributes — SlotType](https://htcondor.readthedocs.io/en/latest/classad-attributes/machine-classad-attributes.html#SlotType) |
| `Cpus` | int | CPUs in this slot | For Partitionable: remaining available CPUs; for Dynamic/Static: allocated CPUs | [Machine ClassAd Attributes — Cpus](https://htcondor.readthedocs.io/en/latest/classad-attributes/machine-classad-attributes.html#Cpus) |
| `Memory` | int | Memory in this slot (MB) | Same semantics as Cpus by slot type | [Machine ClassAd Attributes — Memory](https://htcondor.readthedocs.io/en/latest/classad-attributes/machine-classad-attributes.html#Memory) |
| `Disk` | int | Disk available (KB) | Pool disk capacity | [Machine ClassAd Attributes — Disk](https://htcondor.readthedocs.io/en/latest/classad-attributes/machine-classad-attributes.html#Disk) |

| `TotalSlotMemory` | int | Total memory on the machine (MB) | Used as total capacity for Partitionable slots | [Machine ClassAd Attributes — TotalSlotMemory](https://htcondor.readthedocs.io/en/latest/classad-attributes/machine-classad-attributes.html#TotalSlotMemory) |
| `TotalLoadAvg` | float | Machine load average | Pool load indicator | [Machine ClassAd Attributes — TotalLoadAvg](https://htcondor.readthedocs.io/en/latest/classad-attributes/machine-classad-attributes.html#TotalLoadAvg) |
| `Activity` | str | Slot activity state | Determines idle vs. busy classification | [Machine ClassAd Attributes — Activity](https://htcondor.readthedocs.io/en/latest/classad-attributes/machine-classad-attributes.html#Activity) |

| `Arch` | str | CPU architecture (e.g., "X86_64") | Pool heterogeneity tracking | [Machine ClassAd Attributes — Arch](https://htcondor.readthedocs.io/en/latest/classad-attributes/machine-classad-attributes.html#Arch) |
| `GPUs` | int | Number of GPUs in slot | GPU resource tracking | [Machine ClassAd Attributes — GPUs](https://htcondor.readthedocs.io/en/latest/classad-attributes/machine-classad-attributes.html#GPUs) |

### Slot Activity Values

| Value | Meaning |
|---|---|
| `Idle` | Slot is available for jobs |
| `Busy` | Slot is running a job |
| `Suspended` | Job suspended by policy |
| `Vacating` | Job being evicted |
| `Benchmarking` | Running benchmarks |
| `Retiring` | Slot draining for maintenance |

### Slot State Values

| Value | Meaning |
|---|---|
| `Unclaimed` | No job matched; available for matchmaking |
| `Claimed` | Matched to a job; executing |
| `Owner` | Machine owner is active; not available |
| `Preempting` | Preempting a running job |
| `Drained` | Administratively drained |

### Partitionable Slot Handling

This is a critical detail: Partitionable slots in HTCondor represent the whole machine's available resources. When jobs are matched, Dynamic child slots are created from the Partitionable parent. The `_summarize_slots()` function handles this correctly:

- **Partitionable parent slots:** Counted for total capacity (`TotalSlotCpus`/`TotalSlotMemory`) and idle capacity (`Cpus`/`Memory` = remaining unallocated). **Not** counted as `total_slots`.
- **Dynamic child slots:** Counted as `total_slots` and `claimed_slots`. Their `Cpus`/`Memory` represent what's allocated to the running job. **Not** double-counted with parent.
- **Static slots:** Counted normally based on `State` and `Activity`.

### Output: `PoolSummary` dataclass

| Field | Type | Description |
|---|---|---|
| `total_slots` | int | Count of non-Partitionable slots |
| `idle_slots` | int | Count of Unclaimed or Idle slots |
| `claimed_slots` | int | Count of Claimed slots |
| `other_slots` | int | Count of Preempting/Vacating/etc. |
| `total_cpus` | int | Sum of all CPUs across pool |
| `idle_cpus` | int | Sum of CPUs in idle/available slots |
| `total_memory_mb` | int | Sum of all memory (MB) |
| `idle_memory_mb` | int | Sum of memory in idle/available slots |
| `total_disk_kb` | int | Sum of all disk (KB) |
| `total_gpus` | int | Sum of all GPUs |
| `idle_gpus` | int | Sum of GPUs in idle/available slots |
| `machines` | int | Count of unique machine hostnames |
| `load_avg` | float | Sum of TotalLoadAvg across machines |
| `os_arch` | str | Comma-separated list of OS/Arch combos |
| `slots` | List[Dict] | Raw slot ClassAds (not serialized to JSONL) |

### JSONL Event Fingerprinting

Pool status fingerprint: `"total_slots:claimed_slots:idle_slots:total_cpus:idle_cpus"`. A new `pool_status` event is emitted only when this string changes.

## 6. HTCondor User Priority (condor_userprio) — Fair-Share Standing

**Source:** HTCondor Negotiator daemon
**Access method:** Subprocess only (`condor_userprio -long`)
**When polled:** On-demand (only via `--why-idle`)
**Module:** `why_idle.py` — `_query_userprio()` → `List[Dict]`
**Why it matters:** When jobs are idle despite available resources, priority may be the cause. HTCondor's fair-share negotiator assigns resources to users with better (lower) effective priority first. This data answers "am I being starved by other users?"

### Subprocess Command

```bash
condor_userprio -long [-pool <HOST>]
```

### Output Format

The `-long` flag produces ClassAd-style output: `key = value` pairs separated by blank lines (one block per user). The parser handles both string and numeric values.

### Fields Parsed Per User

| Field | Type | Description | Why needed |
|---|---|---|---|
| `Name` | str | Username (format: `user@domain` or `user@uid_domain`) | Matched against workflow owner (`user@` prefix) to identify current user |
| `EffectivePriority` (or `Priority`) | float | Current effective priority factor | **Lower = better.** Users with lower effective priority are matched first by the negotiator. This is the key metric |
| `RealPriority` (or `PriorityFactor`) | float | Base priority factor before fair-share adjustment | Shows the starting priority before usage-based adjustment |
| `ResourcesUsed` (or `WeightedResourcesUsed`) | float | Cumulative weighted resource usage | Shows resource consumption history; higher usage degrades effective priority |

### Analysis Logic

The `_analyze()` function in `why_idle.py`:
1. Finds the current user's entry by matching `Name.startswith(f"{current_user}@")`
2. Collects other users' entries for comparison
3. Counts users with better (lower) effective priority
4. Generates findings: "Your effective priority is X — N user(s) have better priority"
5. Generates suggestions about fair-share improvement strategies

### Reference Documentation

- **condor_userprio manual:** [condor_userprio man page](https://htcondor.readthedocs.io/en/latest/man-pages/condor_userprio.html)
- **Fair-share scheduling:** [HTCondor Admin Guide — User Priorities and Negotiation](https://htcondor.readthedocs.io/en/latest/admin-manual/user-priorities-negotiation.html)
- **Local reference:** `~/.claude/references/htcondor.md` — Section: "User Priorities"

## 7. HTCondor Negotiator (condor_status -negotiator) — Matchmaking Timing

**Source:** HTCondor Negotiator daemon
**Access method:** Subprocess only (`condor_status -negotiator -json`)
**When polled:** On-demand (only via `--why-idle`)
**Module:** `why_idle.py` — `_query_negotiator()` → `Optional[Dict]`
**Why it matters:** The negotiator runs in cycles (typically every 60-300 seconds). If the last cycle produced zero matches or took an unusually long time, jobs may be idle simply because the negotiator hasn't run yet or is overloaded.

### Subprocess Command

```bash
condor_status -negotiator -json -attributes LastNegotiationCycleDuration0,LastNegotiationCycleMatches0,LastNegotiationCycleDuration1,LastNegotiationCycleMatches1 [-pool <HOST>]
```

### Attributes

| Attribute | Type | Description | Why needed |
|---|---|---|---|
| `LastNegotiationCycleDuration0` | float | Duration of the most recent negotiation cycle (seconds) | Cycles > 60s indicate an overloaded negotiator |
| `LastNegotiationCycleMatches0` | int | Number of job-to-slot matches in the most recent cycle | 0 matches with idle jobs = requirement mismatch or priority starvation |
| `LastNegotiationCycleDuration1` | float | Duration of the second-most-recent cycle | Trend detection |
| `LastNegotiationCycleMatches1` | int | Matches in the second-most-recent cycle | Trend detection |

### Analysis Logic

- Cycles > 60 seconds trigger a suggestion to contact the pool admin
- Zero matches with idle jobs triggers: "Zero matches suggests a requirements mismatch, insufficient resources, or a negotiator configuration issue"

### Reference Documentation

- **Negotiator ClassAd attributes:** [Negotiator ClassAd Attributes](https://htcondor.readthedocs.io/en/latest/classad-attributes/negotiator-classad-attributes.html)
- **Negotiation cycle configuration:** `NEGOTIATOR_INTERVAL` (default 60s) in [HTCondor Configuration](https://htcondor.readthedocs.io/en/latest/admin-manual/configuration-macros.html)
- **Local reference:** `~/.claude/references/htcondor.md` — Section: "Negotiation and Matchmaking"

## 8. Kickstart Output Files (.out.000) — Job-Level Forensics

**Source:** Files in submit directory, written by Pegasus kickstart wrapper
**Typical path:** `<submit_dir>/00/00/<exec_job_id>.out.000` (or custom `stdout_file` path from `job_instance` table)
**When read:** On-demand — only for held or failed jobs during diagnostics
**Module:** `diagnostics.py` — `parse_kickstart_output()` → `Optional[KickstartInfo]`
**Why it matters:** When a job fails, the exit code alone often isn't enough to diagnose the root cause. The kickstart output file contains the job's stdout, stderr, executable path, arguments, duration, and transformation name — providing the full forensic context needed for actionable diagnostics.

### File Format

YAML document written by Pegasus's kickstart wrapper. Top-level structure:
```yaml
- mainjob:
    executable:
      file_name: "/path/to/executable"
    argument_vector: ["arg1", "arg2"]
    duration: 12.345
    status:
      regular_exitcode: 1
  files:
    stdout:
      data: "stdout content here"
    stderr:
      data: "stderr content here"
  transformation: "preprocess"
```

### Parsed Fields (`KickstartInfo` dataclass)

| Field | Type | Source path in YAML | Why needed |
|---|---|---|---|
| `exitcode` | Optional[int] | `mainjob.status.regular_exitcode` | More reliable than raw wait-status from stampede.db |
| `stdout` | str | `files.stdout.data` | Job output for diagnostics |
| `stderr` | str | `files.stderr.data` | Primary error information; drives pattern matching |
| `executable` | str | `mainjob.executable.file_name` | Shown in diagnostic output |
| `argv` | List[str] | `mainjob.argument_vector` | Shown in diagnostic output |
| `duration` | Optional[float] | `mainjob.duration` (seconds) | Job runtime independent of HTCondor overhead |
| `transformation` | str | `transformation` (top-level) | Transformation name for context |
| `stderr_analysis` | Optional[StderrAnalysis] | Derived from `stderr` field | Structured pattern-matching results |

### Stderr Pattern Analysis (`StderrAnalysis` dataclass)

The `_analyze_stderr()` function runs regex patterns against the stderr content to extract structured findings:

| Field | Type | Regex Pattern | What it detects |
|---|---|---|---|
| `missing_files` | List[str] | `Expected local file does not exist:\s*(.+)` | Input files that were expected but not present on the execute node |
| `transfer_attempts` | int | `Starting transfers - attempt (\d+)` | Max transfer retry count; > 1 indicates transient failures |
| `transfers_failed` | bool | `Some transfers failed` | Explicit transfer failure marker |
| `integrity_error` | bool | `integrity verification failed` | Checksum mismatch on transferred files |
| `permission_denied` | List[str] | `Permission denied.*?:\s*(.+)` | Files/directories with access issues |
| `no_space` | bool | `No space left on device` | Disk exhaustion on execute node |
| `connection_error` | bool | `Connection refused\|Connection timed out\|Network is unreachable` | Network issues during file transfer |

### Hold Reason Pattern Matching

For held jobs, `diagnose_held_job()` matches `HoldReason` (from condor_q) against 10 compiled regex patterns in `_HOLD_PATTERNS`:

| Pattern | Summary | Example HoldReason |
|---|---|---|
| Transfer output + No such file | Output file not created | "Transfer output files failure: No such file or directory" |
| Transfer output | Generic output transfer failure | "Transfer output files failure" |
| Transfer input | Input transfer failure | "Transfer input files failure" |
| memory/MEMORY_EXCEEDED/OOM | Memory limit exceeded | "Job exceeded memory limit" |
| disk/DISK_EXCEEDED | Disk limit exceeded | "Job exceeded disk limit" |
| wall.?time/TIME_EXCEEDED | Wall time exceeded | "Job exceeded wall time" |
| credential/proxy/certificate | Credential issue | "Cannot access myproxy server" |
| SHADOW_EXCEPTION | Condor shadow error | "SHADOW_EXCEPTION at execute node" |
| DNS/hostname | DNS resolution failure | "Cannot resolve hostname" |
| docker/container/singularity | Container runtime error | "Docker container failed to start" |
| periodic.*hold/policy | Policy-based hold | "Job held by periodic hold expression" |

### Exit Code Suggestions

For failed jobs without specific stderr patterns, `_FAILURE_SUGGESTIONS` maps exit codes to remediation guidance:

| Exit Code | Meaning |
|---|---|
| 1 | General error |
| 2 | Shell misuse / missing arguments |
| 126 | Command not executable (permissions) |
| 127 | Command not found |
| 137 | Killed by SIGKILL (OOM killer) |
| 139 | Segmentation fault |

### Public API

```python
collect_diagnostics(
    held_jobs: List[JobRecord],      # from WorkflowSnapshot.held_jobs()
    failed_jobs: List[JobRecord],    # from WorkflowSnapshot.failed_jobs()
    condor_jobs: Optional[List[Dict]], # from condor_q (for HoldReason)
    submit_dir: Optional[Path],       # for locating .out.000 files
) -> List[Diagnostic]
```

Output `Diagnostic` dataclass: `job_name`, `severity` ("held"/"failed"/"warning"), `summary`, `reason`, `suggestions` (List[str])

### Reference Documentation

- **Kickstart/PegasusLite output format:** [Pegasus User Guide — Kickstart](https://pegasus.isi.edu/documentation/reference-guide/kickstart.html)
- **Local reference:** `~/.claude/references/pegasus.md` — Section: "Monitoring and Debugging"

## 9. JSONL Event Log (output) — workflow-events.jsonl

**Output file:** `<submit_dir>/workflow-events.jsonl` (default)
**Written by:** `event_log.py` — `EventLogger` class
**When written:** Each poll cycle in `--serve` or `--log` modes
**Purpose:** Persistent, replayable record of the workflow's lifecycle. Used by `--replay` for offline review and `--remote` for SSH-based monitoring.

### Event Types and Schemas

#### `workflow_start` — Emitted once (header)

```json
{
  "event_type": "workflow_start",
  "timestamp": 1719500000.0,
  "wf_uuid": "abc-123-def",
  "dax_label": "diamond",
  "user": "stealey",
  "planner_version": "5.0.7",
  "submit_dir": "/path/to/run0001",
  "wf_start": 1719499990.0
}
```

**Data sources:** `WorkflowInfo` (braindump.yml), `get_workflow_times()` (stampede.db)

#### `jobs_init` — Emitted once (first snapshot)

```json
{
  "event_type": "jobs_init",
  "timestamp": 1719500001.0,
  "wf_uuid": "abc-123-def",
  "total_jobs": 12,
  "jobs": [
    {"job_id": 1, "exec_job_id": "create_dir_diamond_0_local", "type_desc": "create-dir"},
    {"job_id": 2, "exec_job_id": "preprocess_ID0000001", "type_desc": "compute", "transformation": "preprocess", "task_argv": "-i input.txt"},

  ]
}
```

**Data sources:** `WorkflowSnapshot.jobs` (stampede.db)

#### `workflow_state` — On workflow state change

```json
{
  "event_type": "workflow_state",
  "timestamp": 1719500002.0,
  "wf_uuid": "abc-123-def",
  "state": "WORKFLOW_STARTED",
  "status": null,
  "wf_start": 1719499990.0
}
```

**Change detection:** Only emitted when `wf_state` differs from last recorded state.
**Data sources:** `WorkflowSnapshot.wf_state` (stampede.db `workflowstate` table)

#### `job_state` — Per job state transition

```json
{
  "event_type": "job_state",
  "timestamp": 1719500010.5,
  "wf_uuid": "abc-123-def",
  "exec_job_id": "preprocess_ID0000001",
  "type_desc": "compute",
  "state": "EXECUTE",
```


**Change detection:** Fingerprint based on `(ClusterId, JobStatus, HoldReason, RemoteHost, BytesSent, BytesRecvd)` per job. Only emitted when fingerprint changes.
**Data sources:** condor_q ClassAds (full attribute set)

#### `htcondor_history` — When new completed jobs appear

```json
{
  "event_type": "htcondor_history",
  "timestamp": 1719500030.0,
```

**Change detection:** Fingerprint based on set of `ClusterId` values. Emitted when new cluster IDs appear.

```json
{
```

**Change detection:** Fingerprint: `"total_slots:claimed_slots:idle_slots:total_cpus:idle_cpus"`. Emitted when fingerprint changes.

```json
{
  "event_type": "workflow_stats",
  "timestamp": 1719500100.0,
```

### Resume Logic

The `EventLogger` supports stop/restart continuity:
1. On initialization, scans existing JSONL file for the same `wf_uuid`
2. Recovers: `high_water_ts`, `last_wf_state`, `jobs_init_emitted`, condor/history/pool fingerprints
3. Truncates any trailing `workflow_end` event (from previous session) for seamless appending
4. Result: stop/restart produces one continuous event series, no duplicates

## 10. Derived Statistics — WorkflowStats

**Module:** `stats.py` — `compute_workflow_stats()` → `WorkflowStats` dataclass
**When computed:** At workflow completion (before `workflow_end` event)
**Data sources:** `WorkflowSnapshot` (stampede.db) + `condor_history` (optional) + `PoolSummary` (optional)
**Why it matters:** Provides a comprehensive end-of-workflow performance summary: how efficiently resources were used, where bottlenecks occurred, and how the workflow could be optimized.

### Fields

#### Job Counts (from `WorkflowSnapshot`)

| Field | Derivation |
|---|---|
| `total_jobs` | `snapshot.total_jobs()` |
| `compute_jobs` | `len(snapshot.compute_jobs())` |
| `infra_jobs` | `total_jobs - compute_jobs` |
| `succeeded` | `snapshot.done_count()` |
| `failed` | `snapshot.failed_count()` |
| `held` | `snapshot.held_count()` |

#### Timing (from `WorkflowSnapshot`)

| Field | Derivation | Why |
|---|---|---|
| `wall_time` | `snapshot.elapsed` | Total workflow wall-clock time |
| `total_compute_time` | Sum of all compute job durations | Total CPU-seconds consumed |
| `parallelism` | `total_compute_time / wall_time` | Effective parallelism factor; > 1 = multiple jobs ran concurrently |

#### Duration Distribution (compute jobs only, from `WorkflowSnapshot`)

| Field | Derivation |
|---|---|
| `dur_min`, `dur_max`, `dur_mean`, `dur_median` | Statistics over compute job durations |
| `longest_job_name`, `shortest_job_name` | Job names with max/min duration |

#### Memory (from stampede.db `invocation` table via `WorkflowSnapshot`)

| Field | Derivation |
|---|---|
| `peak_maxrss_kb` | Max `maxrss` across all compute jobs |
| `peak_maxrss_job` | Job name with highest `maxrss` |
| `mean_maxrss_kb` | Mean `maxrss` across all compute jobs |

#### HTCondor Efficiency (from `condor_history`, optional)

| Field | Derivation |
|---|---|
| `cpu_eff_min`, `cpu_eff_max`, `cpu_eff_mean` | Min/max/mean of `cpu_efficiency()` across history records |
| `mem_eff_min`, `mem_eff_max`, `mem_eff_mean` | Min/max/mean of `memory_efficiency()` across history records |

#### Queue Wait (from `condor_history`, optional)

| Field | Derivation |
|---|---|
| `wait_min`, `wait_max`, `wait_mean` | Min/max/mean of `queue_wait_seconds()` = `JobStartDate - QDate` |

#### Aggregates (from `condor_history`, optional)

| Field | Derivation |
|---|---|
| `cpu_seconds` | Sum of `RemoteUserCpu` across all history records |
| `transfer_bytes` | Sum of `BytesSent + BytesRecvd` across all history records |
| `retry_count` | Sum of `(NumJobStarts - 1)` for jobs with retries |
| `hosts` | List of execution hosts with job counts, e.g., `["machine1 (4)", "machine2 (2)"]` |

| `pool_total_cpus` | `pool_status.total_cpus` |

## 11. Consumption Modes

| Mode | CLI Flag | Data Sources | Poll Frequency |
|---|---|---|---|
| **Live TUI** | *(default)* | stampede.db + condor_q + condor_history + condor_status | Base: `--interval` (2s); history: ~10s; pool: ~15s |
| **One-shot** | `--once` | stampede.db + condor_q | Single poll |
| **Server/Daemon** | `--serve` | Same as Live TUI | Same, writes JSONL + PID file |
| **Remote SSH** | `--remote user@host:/path` | JSONL via `ssh tail -c +offset` | Every `--sync-interval` (5s) |
| **Replay** | `--replay FILE` | JSONL file | Frame-based, `--speed` controlled |
| **Why-Idle** | `--why-idle` | stampede.db + condor_q (idle only) + condor_status + condor_userprio + condor_status -negotiator | Single poll, prints and exits |
